Frequently asked questions for SAP employees

May 20, 2014 by Ingo

This FAQ tries to answer common questions SAP employees have when they want to or are already in the process of publishing an Open Source project on sap.github.io. The page is intended to help get an overview of what is needed by SAP as a company and/or is common best practices for open source projects.

- What do I need to do to open source my project

If you are an SAP group or employee and are interested in making your SAP project open source you have to go through the SAP Outbound Open Source Process. The process is described on the intranet Open Source pages of SAP @ https://community.wdf.sap.corp/groups/open-source, accessible from within the SAP corporate network only.

- Who do I contact for a repository?

The complete list of administrators for sap.github.io can be found @ https://github.com/orgs/SAP/teams/owners provided that you are logged-in into GitHub. While you are able to send either one of them a request in most cases you want to contact:

  • Matthias for SAP Hana Cloud based projects
  • Ingo for everything else
Also if you have questions in general you can send an email to the SAP Developer Center email to .

- What do I have to have, can have, or should not have in my repository?

We do not request much. Outside of keeping your repository clean and understandable for people we would like every repository owner to have:

  • A Readme.md file describing the project (HANA node.js driver example).
  • The license agreement for the project approved through the Outbound Open Source Process has to be inside the top folder of your repository and be named LICENSE.
  • Include all required notices for all third party open source components that are included in the project (this does not apply to dependencies, only third party components that can be downloaded from the project).

Outside of that we do not have any other requirements. What you could think of is:

  • Have the admins activate the wiki pages for your repository
  • Put up pages for support on StackOverflow (OpenUI5 example) or the SAP Coummunity Network forums (OpenUI5 example)
  • Maintain pages for your project on GitHub like the OpenUI5 have done it with the gh-pages capability of GitHub.
Finally what we do not want you to do is:
  • Never store libraries your project depends on and that are not part of your project inside your repository. Either explain how to assemble what is needed in the Readme.md or on help pages.Some languages allow you to make references to the libraries in build files that are resolved during build time. If the language or environment you use has this option consider using it.
  • Never add any source or compiled code of SAP that is not part of your project inside your repository. Also make sure that all files inside your repository are subject to or compliant with the project's license agreement (store in the top folder as noted above).

- Where do I put the open source license?

The complete license agreement of the respective license should be inside the top folder of your repository and the file should be named LICENSE. Many open source licenses have short version headers for every source code file inside the project and you may want to consider adding this header to the source files.

- What name can I give to my repository?

In the past we used to put projects into categories like cloud, mobile, data and UI and organized the projects in that manner on the main page of sap.github.io. To support this structure, we had started a naming convention that a repository name starts with the category name. But since we have been receiving projects that do not fit these categories, we decided to drop this convention altogether. At this point the simple rule is that the name should be descriptive should not be used already at sap.github.io.

- Can I make use of my existing user account at github?

Yes, you can use an existing GitHub user to participate or manage an open source project on sap.github.io. We do suggest that you add your sap corporate email address to that user for communication purposes and make that email address the primary GitHub email if possible.

- What email address should I use?

SAP suggest that you use your corporate email address as the default email as the primary communication email inside your GitHub user you use to work on an Open Source project hosted on sap.github.io

- How should I handle pull requests from non-sap employees?

SAP requires everyone contributing to an SAP led open source project hosted on sap.github.io to sign a Individual Contributor License Agreement (ICLA). If the contributor is made on behalf of a corporation then a Corporate Contributor License Agreement should be signed by the corporation, however, this does not remove the requirement for every individual contributor representing the corporation to sign an ICLA. The project lead is responsible for obtaining signatures and storing the documents. Once signatures are obtained pull requests can be accepted.
